Los 6 agentes de IA

El bot no es un solo modelo gigante: son 6 agentes especializados. Uno decide a quién mandar el mensaje (Router), 5 hacen el trabajo, y un Formatting Agent post-procesa el output antes de mandarlo a WhatsApp.

AgenteModeloFunciónTools
Router gpt-5-mini Lee el mensaje + contexto y decide a cuál de los 5 sub-agents mandar —
Sub-Agent Confirmar gpt-5 Confirma turnos cuando el paciente responde a un recordatorio (sí/dale/👍) 7 tools Dentalink + Supabase
Sub-Agent Cancelar gpt-5 Cancela o reprograma turnos. Detecta multi-fichas (familias) y resuelve por contexto 8 tools incluyendo cancelar_turno + ver_turnos
Sub-Agent Agendar gpt-5 Agenda turnos nuevos. Identifica al paciente, ofrece slots, reserva en Dentalink 7 tools incluyendo buscar_horarios + reservar_turno
Sub-Agent Urgencia gpt-5-mini Detecta dolor / aparato roto / sangrado → deriva al grupo escalar_a_secretaria
Sub-Agent General gpt-5-mini Info canned: precio, horarios, dirección, alias bancario. Y silenciar lo no operativo KB + escalar
Formatting Agent gpt-5 Recibe el texto del sub-agent y lo deja listo para WhatsApp (formato hs, split, emojis) —
Por qué dividir en 6 agentes y no uno solo: cada sub-agente tiene un system prompt enfocado en SU función. Eso reduce alucinaciones (Confirmar no sabe agendar, así que no inventa horarios), permite usar modelos más baratos donde no hace falta razonar tools (mini en Router y General), y hace que cada bug se aísle al agente correspondiente.

Las 9 herramientas (qué puede hacer)

Los sub-agents no escriben respuestas mágicas: para actuar sobre el mundo real necesitan tools. Cada tool es una función concreta que llama a un sistema externo.

🔍 buscar_paciente_dentalink read

GET Dentalink /pacientes

Busca al paciente por los últimos 10 dígitos del celular o por apellido. Detecta multi-fichas (familias con un mismo número).

📅 buscar_horarios read

GET Dentalink /citas

Busca slots disponibles para una fecha. Lee y solo informa — no agenda nada por sí solo.

📋 ver_turnos_paciente read

GET Dentalink /citas?id_paciente=X

Trae los turnos de un paciente. Se usa para evitar doble-booking y para resolver "el de julio".

✅ reservar_turno write

POST Dentalink /citas

Crea el turno en Dentalink (estado PRE-reservado). Solo después de read-back y confirmación del paciente.

📌 confirmar_turno write

PUT Dentalink /citas/{id} estado=18

Marca el turno como "Confirmado por WhatsApp". Lo llama Sub-Agent Confirmar tras un recordatorio.

❌ cancelar_turno write

PUT Dentalink /citas/{id} estado=1

Anula el turno en Dentalink. Body solo acepta {"id_estado": 1}, otros params devuelven 400.

👤 crear_paciente_dentalink write

POST Dentalink /pacientes

Crea ficha nueva. Solo si las búsquedas fallaron Y el paciente confirmó nombre + DNI.

📚 buscar_conocimiento read

Vector search en Supabase KB

RAG sobre la base de conocimiento de la clínica (34 entradas: tratamientos, FAQ, horarios). Solo lo usa el Sub-Agent General.

📢 escalar_a_secretaria write

Webhook → Evolution → grupo WhatsApp

Manda un resumen del caso al grupo de Iri + Dra + Lucas para que un humano se encargue.

Las 3 capas de seguridad

Después del incidente de Mariela (09/05), pusimos defensa en profundidad. Si una capa falla, la siguiente atrapa el problema.

graph LR O[Output del sub-agent] --> L1[Capa 1: Prompt rules
R0, banlist en system] L1 --> L2[Capa 2: Banlist Validator
regex post-output] L2 --> L3[Capa 3: Label humano auto
si banlist hit, escalar] L3 --> S[Send WhatsApp] style L1 fill:#ddd6fe,stroke:#5b21b6 style L2 fill:#fee2e2,stroke:#991b1b style L3 fill:#fef3c7,stroke:#92400e

Capa 1 · Prompt rules

Cada sub-agent tiene en su system prompt la regla R0: NO inventes, NO diagnostiques, NO digas "venite". El LLM lo respeta el 95% del tiempo.

Capa 2 · Banlist regex

Después del Formatting, un código JS escanea el output con ~30 regex de frases prohibidas. Si matchea, el output se descarta y el caso se escala.

Capa 3 · Label humano auto

Si la Capa 2 atrapa algo, además de no enviarlo se marca al paciente con label humano en Chatwoot. El bot queda silenciado para ese paciente hasta que la Dra lo desbloquee.

🚧 En desarrollo · Banlist Agent

Sub-agent gpt-5-nano que valida el output con contexto. Captura paráfrasis que el regex no agarra ("diríjase al consultorio"). Arrancará en shadow mode para validar antes de darle autoridad.

El incidente Mariela y por qué importan estas capas: el 09/05/2026 una mamá real recibió del bot "guardá la pieza, venite ahora mismo a Balcarce 37, te esperamos" un sábado con clínica cerrada. La Dra alcanzó a apagar el bot a tiempo. Root cause triple: kill-switch usaba phone del destinatario en lugar del emisor, system prompt enseñaba "Te esperamos" como ejemplo (que contradice la regla R22), y el router no veía memoria. Las 3 capas actuales hacen que un mismo bug nunca pueda repetirse.
